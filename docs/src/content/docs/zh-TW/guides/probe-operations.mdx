## 探測器生命週期

將探測器視為觀察網路的一個視角。探測器應放在量測視角真正重要的位置，例如邊緣區域、實驗室網路、社群節點、私人伺服器或供應商邊界。

保持探測器身分穩定。當探測器長期代表相同位置時，歷史延遲、DNS 與路由資料最有價值。

## 檢查

先從少量、能回答維運問題的檢查開始。

- 使用延遲檢查，從單一視角找出封包遺失與 jitter。
- 使用 DNS 檢查，比較 resolver 行為與 authoritative response。
- 使用路由檢查，找出一般 uptime check 容易漏掉的路徑變化。

## 結果流程

產品應用程式應將探測結果呈現為來自 Go API 的 server state。說明文件網站應解釋此行為並連結至產生的 API 範例，不應重複產品程式碼。

:::tip 新增 check type 時，請先更新後端合約、重新產生 `docs/public/openapi.json`，再於此處記錄其維運意義。 :::

## 維運習慣

記錄每個探測器存在的原因。像是 `home fiber edge`、`university lab` 或 `regional VPS` 的簡短備註，可以讓未來的比較更容易解讀。

基礎架構變更後應重新檢查探測器。更換主機、搬移區域或變更上游供應商，都可能讓舊量測失去代表性。
