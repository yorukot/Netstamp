## 探針生命週期

把探針視為觀察網路的一個視角。將探針放在量測觀點重要的位置，例如邊緣區域、實驗室網路、社群節點、私人伺服器或服務供應商的邊界。

維持探針身分穩定。當一個探針長期代表同一個地點時，歷史的延遲、DNS 與路由資料最有參考價值。

## 檢查

從一小組能回答營運問題的檢查開始。

- 用延遲檢查從單一視角觀察封包遺失與抖動。
- 用 DNS 檢查比較解析器行為與權威回應。
- 用路由檢查捕捉一般可用性檢查會錯過的路徑變化。

## 結果流向

產品應用應把探針結果呈現為來自 Go API 的伺服器狀態。文件網站則應說明這個行為並連到產生的 API 範例，而不是複製產品程式碼。

:::tip 新增檢查類型時，先更新後端合約、重新產生 `docs/public/openapi.json`，再在這裡記錄它的營運意義。:::

## 營運習慣

記錄每個探針存在的理由。像 `home fiber edge`、`university lab` 或 `regional VPS` 這樣的簡短註記，能讓未來的比較更容易解讀。

在基礎設施變動後檢視探針。更換主機、搬遷區域與更改上游供應商，都可能讓舊的量測產生誤導。
