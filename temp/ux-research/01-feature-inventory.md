# Netstamp 已實作功能盤點

## 方法說明

- 依 repo 指示先讀取根目錄 `AGENTS.md`，再讀取 `server/AGENTS.md`、`api/AGENTS.md`、`web/AGENTS.md` 與實際存在的 `DESIGN.md`。本輪未找到 lowercase `design.md`；未追蹤檔 `ux-design.md` 未作為 evidence。
- 掃描範圍包含 web routes/pages/components、TypeSpec API schema、Go HTTP handlers/services、SQL migrations/query、agent CLI/runtime、docs/deployment/config、Storybook/UI package、測試檔與 TODO。
- 判定原則：只列出有 route、schema、handler、migration、component、test 或 config 佐證的功能。若只在品牌名稱或泛用詞中出現，不推論成產品功能。
- 高階判讀：Netstamp 目前已實作的產品面明確是 self-hosted network observability，核心物件是 projects、probes、checks、assignments、results、alerts、status pages。未找到 document/stamp/certificate/verification 作為文件驗證或數位簽章產品的證據。

## 主要 UX 風險摘要

- 產品名與功能認知：`Netstamp` 容易讓人聯想到 document timestamp/stamp/certificate，但目前實作是 network observability。若外部頁面或銷售文案不清楚，會造成期待落差。
- First-run 摩擦：onboarding 會帶到 probe wizard，但實際成功需要 root Linux/systemd、下載 binary、設定 controller URL/probe secret 並等 heartbeat。這是核心 activation risk。
- 資訊密度高：Checks selector、Insight controls、Alert rules、Status page elements 都是工程向複雜表單，空狀態與錯誤狀態需要更明確地告訴使用者下一步。
- 權限與 read-only：demo/read-only、role permission、feature flags 同時存在；若 UI 顯示可點但 mutation 才被擋，使用者會覺得不一致。
- Alerts 未完成面：alerting 已可用，但 beta、traceroute disabled、incident 無手動處理 action，可能影響營運團隊採用信心。
- Invite UX：UI 顯示 `Invite sent`，但 evidence 未看到 email delivery；若只支援已存在帳號，應在 UI 明講，否則會被理解成真的寄信邀請。
- Error feedback 混合：problem details、toast、table emptyLabel、plain 500 並存；對排查 probe/install/runtime failures 不夠集中。
- Frontend regression 風險：核心 UX 流程很多，但未找到 web feature tests；複雜表單和 route guards 特別容易在後續修改中退化。
